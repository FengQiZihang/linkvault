# 项目避坑

这里记录了在本项目中踩过的坑，方便之后自己和 AI 别再犯同样的错。

---

## 1. Maven 用的是项目里的配置，不是系统的

这个项目的 Maven 配置不走系统默认，而是靠项目根目录下的 [.mvn/maven.config](.mvn/maven.config) 来控制的。这个文件里写死了两样东西：

- `settings.xml` 的位置：`/home/hang/workspace/DevTools/maven/current/conf/settings.xml`
- 本地仓库的位置：`/home/hang/workspace/DevTools/maven/current/mvn_repo`

也就是说，只要你在 `backend/linkvault-server` 目录下执行任何 Maven 命令（`mvn test`、`mvn compile`、`mvn spring-boot:run`），Maven 会自动读取这个配置文件，不需要你每次都手动加 `--settings` 和 `-Dmaven.repo.local` 参数。

**怎么排错**：如果哪天发现依赖下不下来、仓库源不对、或者命令结果和你手动跑的不一样，第一件事就是检查这个 `.mvn/maven.config` 文件还在不在、内容有没有被改过。

## 2. Lombok 别一刀切，分情况用

项目里用 Lombok 的时候别图省事全部加 `@Data`，不同类型的类要区别对待：

**数据类（DTO、VO、简单值对象）**：用 `@Getter` + `@Setter` + `@NoArgsConstructor` + `@AllArgsConstructor`。这类只负责装数据，替换掉手写的 getter/setter 最稳。

**注入类（Controller、Service、Interceptor、Config）**：用 `@RequiredArgsConstructor`，前提是你要注入的字段用 `final` 修饰。这样 Lombok 会自动生成构造器，Spring 会自动完成注入。

**先别动的类**：`BusinessException`、`ApiResponse`、`JwtTokenProvider`、`UserContext` 这些，它们要么有自定义构造逻辑，要么有静态工厂方法，要么有继承关系。为省几行代码强行上 Lombok 反而容易出问题，现阶段保持手写更稳妥。

**改造顺序**：如果要继续推进 Lombok 改造，顺序是——先数据类 → 再注入类 → 复杂的留着别动。每改完一类就跑一次 `mvn test`，确认没挂再继续。
