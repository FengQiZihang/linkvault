# 项目避坑

这里记录了在本项目中踩过的坑，方便之后自己和 AI 别再犯同样的错。

---

## 1. Maven 用的是项目里的配置，不是系统的

这个项目的 Maven 配置不走系统默认，而是靠项目根目录下的 [.mvn/maven.config](.mvn/maven.config) 来控制的。这个文件里写死了两样东西：

- `settings.xml` 的位置：`/home/hang/workspace/DevTools/maven/current/conf/settings.xml`
- 本地仓库的位置：`/home/hang/workspace/DevTools/maven/current/mvn_repo`

也就是说，只要你在 `backend/linkvault-server` 目录下执行任何 Maven 命令（`mvn test`、`mvn compile`、`mvn spring-boot:run`），Maven 会自动读取这个配置文件，不需要你每次都手动加 `--settings` 和 `-Dmaven.repo.local` 参数。

**怎么排错**：如果哪天发现依赖下不下来、仓库源不对、或者命令结果和你手动跑的不一样，第一件事就是检查这个 `.mvn/maven.config` 文件还在不在、内容有没有被改过。

## 2. Lombok 别一刀切，分情况用

项目里用 Lombok 的时候别图省事全部加 `@Data`，不同类型的类要区别对待：

**数据类（DTO、VO、简单值对象）**：用 `@Getter` + `@Setter` + `@NoArgsConstructor` + `@AllArgsConstructor`。这类只负责装数据，替换掉手写的 getter/setter 最稳。

**注入类（Controller、Service、Interceptor、Config）**：用 `@RequiredArgsConstructor`，前提是你要注入的字段用 `final` 修饰。这样 Lombok 会自动生成构造器，Spring 会自动完成注入。

**先别动的类**：`BusinessException`、`ApiResponse`、`JwtTokenProvider`、`UserContext` 这些，它们要么有自定义构造逻辑，要么有静态工厂方法，要么有继承关系。为省几行代码强行上 Lombok 反而容易出问题，现阶段保持手写更稳妥。

**改造顺序**：如果要继续推进 Lombok 改造，顺序是——先数据类 → 再注入类 → 复杂的留着别动。每改完一类就跑一次 `mvn test`，确认没挂再继续。

## 3. IDEA 的 Database 面板连不上 WSL 里的 MySQL

项目代码、JDK、Maven、MySQL 全都在 WSL 里。VS Code 用 WSL 开发模式能直接连上 MySQL，但 IDEA 打开同一个 WSL 项目，自带终端已经识别到是 WSL 了，右侧的 Database 面板却死活连不上。

问题出在 MySQL 默认只监听 `127.0.0.1`。IDEA 的 Database 工具跑在 Windows 侧，从 Windows 访问 `127.0.0.1` 指向的是 Windows 本机，不是 WSL。所以要让 MySQL 监听 WSL 的 IP 地址，IDEA 才能从 Windows 侧连过去。

具体操作步骤见 [IDEA连接WSL内MySQL操作记录.md](IDEA连接WSL内MySQL操作记录.md)。

## 4. Run 能启动，但 IDEA Debug 启动失败

现象是：在 `backend/linkvault-server` 目录下执行 `mvn spring-boot:run` 可以正常启动后端，IDEA 里点普通 `Run` 也可以启动，但是点 `Debug` 会失败。控制台里可能出现两类线索。

第一类是 JMX 端口冲突：

```text
JMX connector server communication error
Port already in use
java.net.BindException: Address already in use
```

第二类是 JDWP 调试连接刚连上就断开：

```text
-agentlib:jdwp=transport=dt_socket,address=0.0.0.0:12345,suspend=y,server=y
Listening for transport dt_socket at address: 12345
Connected to the target VM, address: 'localhost:2220', transport: 'socket'
Disconnected from the target VM, address: 'localhost:2220', transport: 'socket'
```

这不是 Maven、JDK 或 Spring Boot 业务代码的问题，而是 Windows 端 IDEA 调试 WSL 里的 Java 进程时，IDEA 注入的 JMX/JDWP 调试端口和 WSL 端口转发链路不稳定。

处理办法：

1. 打开 IDEA 右上角运行配置 `LinkvaultServerApplication`，进入 `Edit Configurations...`。
2. 选中 Spring Boot 配置。
3. 如果配置项里有 `Enable JMX authentication`，点右侧的 `x` 把它移除。
4. 点 `Modify options`，如果菜单里有 `Disable JMX endpoints`，勾上它。
5. 普通开发优先用 `Run` 启动；需要断点调试时，优先用下面的远程调试方式。

如果 VM options 里手动写过下面这类参数，也先删掉或换成不冲突的端口：

```text
-Dcom.sun.management.jmxremote
-Dcom.sun.management.jmxremote.port=1376
-Dcom.sun.management.jmxremote.rmi.port=1376
```

更稳的调试方式：在 WSL 终端手动用 JDWP 启动 Spring Boot，然后让 Windows IDEA 以 `Remote JVM Debug` 的方式 attach 进去。

WSL 终端执行：

```bash
cd /home/hang/workspace/AndroidAPP开发/backend/linkvault-server
mvn spring-boot:run -Dspring-boot.run.jvmArguments="-agentlib:jdwp=transport=dt_socket,server=y,suspend=n,address=*:5005"
```

然后 IDEA 新建 `Remote JVM Debug` 配置：

- Host：优先填 `localhost`
- Port：`5005`

如果 `localhost:5005` 连不上，再在 WSL 里执行 `hostname -I` 查看 WSL IP，把 Host 改成对应的 WSL IP。

快速判断思路：只要普通 `Run` 能启动，而 `Debug` 日志里出现 `jdwp`、`Listening for transport dt_socket`、`Connected` 后马上 `Disconnected`，优先怀疑 IDEA + WSL 的调试端口链路，不要先怀疑 Maven 依赖或业务代码。
