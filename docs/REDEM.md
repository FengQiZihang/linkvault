# Docs REDEM

## 1. 这份文件的作用

本文件用于让 AI 或新接手的人在第一次进入项目时，快速理解 `docs` 目录的结构、各文档的职责，以及阅读顺序。

当前项目状态：

- 前后端历史代码已全部删除
- 现在以 `docs` 和 `prototype` 为基线，重新开始工程实现
- 当前最重要的资产不是代码，而是文档链路是否一致

## 3. 权威文档顺序

当多份文档出现重复描述时，优先级按下面理解：

1. 产品需求与范围：`docs/requirements/需求文档.md`
2. 页面结构与交互：`docs/prototype/产品原型.html`
3. 原型补充说明：`docs/prototype/原型设计说明.md`
4. REST API、DTO/VO、错误码：`docs/engineering/接口文档.md`
5. 数据表、字段、索引、查询来源：`docs/engineering/数据库字段设计.md`
6. 最终可执行建表结果：`docs/sql/001_schema.sql`
7. 工程实现约束：`docs/engineering/技术栈规范与环境基线.md`
8. 开发拆分与实施顺序：`docs/engineering/模块拆分任务规划与目录结构.md`

简单说：

- `需求文档` 决定做不做
- `原型` 决定页面长什么样、怎么跳
- `接口文档` 决定前后端怎么交互
- `数据库字段设计` 和 `001_schema.sql` 决定数据怎么落

## 4. 各目录的作用

### `docs/architecture`

放方法论、协作原则、AI 工作方式，不直接定义业务字段。

- `AI-Native软件开发流程与实践指南.md`
  - 解释为什么这个项目要“先文档、后代码”
- `VibeCoding实践心法.md`
  - 更偏开发哲学和工作原则
- `原型-VO-接口对齐方法论.md`
  - 解释原型、VO、接口三者如何对齐
- `项目背景与AI协作提示.md`
  - 当前项目背景、协作边界、当前阶段说明

### `docs/requirements`

放产品需求基线。

- `需求文档.md`
  - 整个项目最重要的业务边界文档
  - 定义 MVP 范围、主流程、页面范围、规则、验收标准

### `docs/prototype`

放页面原型和原型说明。

- `产品原型.html`
  - 页面结构、交互、跳转的强约束来源
- `原型设计说明.md`
  - 解释页面层级、弹窗类型、交互约定
- `页面截图/`
  - 原型截图，仅用于快速查看，不作为额外规则来源

### `docs/engineering`

放工程实现过程中最关键的“结构化设计文档”。

- `接口文档.md`
  - 后端 REST API 基线
  - 包含路径、鉴权、请求、响应、VO、错误码、页面接口组合
- `openapi.yaml`
  - 从接口文档生成的 OpenAPI 3 文件
  - 用于导入 Apifox
- `数据库字段设计.md`
  - 数据表设计、字段来源、查询支撑、派生字段来源
- `技术栈规范与环境基线.md`
  - 技术选型、前后端实现约束、时间策略、API 约定
- `模块拆分任务规划与目录结构.md`
  - 把项目转成可分阶段开发的任务与模块边界
- `后端代码风格约定-基于sky-take-out项目.md`
  - 后端代码风格参考，不直接定义业务
- `WSL开发环境配置记录.md`
  - 本地开发环境记录，用于排查环境问题

### `docs/sql`

放真正可执行的 SQL 资产。

- `000_local_user.sql`
  - 本地开发用数据库账号初始化脚本
  - 仅在使用管理员账号时执行
- `001_schema.sql`
  - 建库建表脚本
  - 只包含可反复执行的结构初始化，不包含账号授权
- `002_seed_test_data.sql`
  - 初始测试数据

### `docs/testing`

放实验记录、问题记录、测试过程沉淀。

- `Java链接源信息解析过程记录.md`
  - 元信息解析实验过程
- `Java链接源信息解析执行结果.md`
  - 元信息解析样本结果
- `重复导入触发共享链接元信息降级问题记录.md`
  - 已发现问题的专项记录

## 6. AI 开始工作前应先确认的事情

在生成代码、改文档、评审设计前，先确认下面几点：

1. 当前任务属于需求、原型、接口、数据库、工程实现、还是测试记录层。
2. 是否已经有对应的权威文档，不要跳过现有文档直接发明新规则。
3. 如果发现冲突，先判断是哪一层文档错了，再改文档，不要先改代码。
4. 原型里出现的字段、按钮、交互，默认是强约束。
5. 接口文档里的 VO 和错误码，默认是前后端联调基线。
6. 数据库字段设计和 `001_schema.sql` 要保持一致；若不一致，最终以 SQL 可执行结果为落点，但应回填说明文档。

---

## 7. 当前实现方向

当前推荐实现顺序：

1. 先以后端接口和数据库为中心，完成最小可测闭环
2. 使用 `docs/engineering/openapi.yaml` 导入 Apifox 做接口测试
3. 后端稳定后，再按原型逐页落地前端
4. 开发时保持最小单元推进：一次只做一个接口或一个页面子功能

---

## 8. 一句话理解整个 docs

`docs` 不是资料堆，而是这个项目在“没有代码时仍然成立”的工程骨架。

只要以下链路一致，代码就可以稳定重建：

`需求 -> 原型 -> 接口 -> 数据库 -> 工程规范 -> 测试记录`
