# LinkVault MVP 第三步基线文档（初稿）

## 0. 文档目标与适用范围

本文档用于落实《VibeCoding实践心法》第三步：确定技术栈、开发规范与环境基线。

适用范围：

- LinkVault Android MVP 阶段
- 后端、前端（UniApp）、数据库、Android 分享桥接开发
- 人工开发与 AI 协同开发的统一工程约束

约束原则：

- 先保证“保存-备注-标签-回找”主闭环可用，再做扩展能力
- 规范以“可执行、可复现、可验收”为目标，不追求过度设计

---

## 1. 技术栈定稿

### 1.1 总体架构

采用“前后端分离 + Android 壳接入系统分享”的实现方式：

- 客户端：UniApp（Vue3）承担业务页面与交互
- 后端：Spring Boot 提供 REST API
- 数据层：MyBatis-Plus + MySQL
- Android 分享桥接：原生最小能力（接收 `ACTION_SEND` -> 跳转 UniApp 承接页）

### 1.2 技术组件清单

后端：

- Java 21（LTS）
- Spring Boot 3.x
- MyBatis-Plus 3.5.x
- MySQL 8.0
- Maven 3.9+

前端（UniApp）：

- Node.js 22 LTS
- UniApp（Vue3）
- uni-ui（仅使用必要组件，避免重度 UI 依赖）
- HBuilderX（用于 Android 打包）

开发与运行环境：

- Docker + Docker Compose（统一 MySQL 与可选中间件）
- Git（分支协作与版本管理）

### 1.3 选型说明（MVP 视角）

- UniApp 满足“一套业务代码优先完成 Android MVP”的目标，降低首版成本
- Spring Boot + MyBatis-Plus 能快速覆盖账号、收藏、标签、搜索等 CRUD 与关系查询场景
- MySQL 足够支撑当前班级项目规模与 MVP 数据量
- Android 原生桥接只做“系统分享接入”最小闭环，避免过早重度原生开发

## 2. 开发规范基线

### 2.1 仓库结构约定

建议采用以下目录结构（可在落库时微调）：

```text
linkvault/
├── backend/                        # Spring Boot 后端父工程
│   └── linkvault/
│       ├── linkvault-common/       # 公共能力：常量、异常、工具、统一响应对象
│       ├── linkvault-pojo/         # 数据对象：entity/dto/vo/page result
│       └── app-server/             # 应用主模块：controller/service/mapper/config
├── frontend/                       # UniApp 前端
│   └── linkvault-uniapp/
├── android-share/                  # Android 分享桥接最小示例 / 壳工程说明
├── docs/                           # 项目文档
│   ├── sql/                        # SQL 脚本（初始化、数据迁移）
│   └── api/                        # 接口文档（Markdown 或 Swagger 导出）
├── docker/                         # 容器化部署配置
│   └── compose.yml                 # 本地依赖服务一键启动（MySQL, Redis 等）
└── README.md                       # 项目自述文件
```

### 2.2 后端规范（Java）

- 包结构按业务域分层：`controller` / `service` / `service/Impl` / `mapper` / `entity` / `dto` / `vo` / `exception` / `config`
- 后端建议采用多模块：`linkvault-common` / `linkvault-pojo` / `app-server`
- 统一返回结构：`code`、`msg`、`data`（如历史接口使用 `message`，需在网关层或响应封装层兼容）
- 参数校验统一使用 `jakarta.validation`
- 全局异常统一处理，禁止直接向前端暴露堆栈
- 数据库操作优先使用 MyBatis-Plus，复杂查询再补充 XML
- 命名统一小驼峰，布尔字段以 `is/has` 语义命名
- 分页结果统一分页对象（如 `PageResult<T>`），避免在各接口重复定义分页字段

### 2.3 前端规范（UniApp/Vue3）

- 前端代码风格统一使用 Vue3 组合式 API（Composition API）
- 页面与组件默认使用 `<script setup>` 写法
- 新增代码禁止使用 Options API（`data` / `methods` / `computed` 组织方式）
- 页面路由按业务域划分：登录、首页、导入承接、标签管理、搜索、详情、我的
- 组件命名：`PascalCase`，页面目录使用 `kebab-case`
- API 调用统一封装（请求拦截、登录态注入、错误提示统一处理）
- 状态管理保持轻量，优先页面局部状态 + 必要全局状态
- UI 风格与原型保持一致：深色系 + 紫色强调色（可在变量层统一管理）

### 2.4 数据库与 SQL 规范

- 表名使用 `lv_` 前缀（如：`lv_user`、`lv_link`、`lv_tag`、`lv_link_tag`）
- 必备审计字段：`created_at`、`updated_at`
- 所有外键关系在应用层与索引层保证一致性（MVP 可不强制物理外键）
- 索引优先覆盖：用户维度查询、链接去重、标签关系、搜索关键路径
- SQL 脚本统一放在 `docs/sql`，并保持可重复初始化

### 2.5 API 约定

- 接口前缀统一：`/api/v1`
- 鉴权方式：登录后下发 JWT，后续请求通过请求头携带
- 请求头规范：优先 `Authorization: Bearer <token>`，兼容 `token` 头作为过渡
- 分页参数统一：`page`、`pageSize`
- 时间字段统一为 ISO 8601 字符串或毫秒时间戳（二选一，项目内必须一致）
- 错误码分层：通用错误、鉴权错误、业务错误

#### 2.5.1 短信验证码方案（MVP）

- 目标：先保证可登录可联调，再逐步增强风控
- 验证码有效期：`5` 分钟
- 发送频率限制：同手机号 `60` 秒内只能发送 `1` 次
- 当日次数限制：同手机号单日最多 `10` 次（超限后拒绝发送）
- 校验失败限制：连续输错 `5` 次后需重新获取验证码
- 存储要求：验证码仅存服务端（Redis 或数据库临时表），禁止前端缓存明文
- 环境约定：
  - `dev`：允许 Mock 短信通道（固定验证码或控制台输出）
  - `prod`：必须使用真实短信服务并开启限流
- 接口返回：不暴露“号码是否已注册”，统一返回文案，避免被批量探测

#### 2.5.2 JWT 生命周期约定（MVP）

- `access token`：默认 `2` 小时有效（短周期）
- `refresh token`：默认 `30` 天有效（长周期）
- 刷新策略：`access token` 过期后，前端使用 `refresh token` 换新
- 失效策略：
  - 用户主动退出登录：立即使当前设备 token 失效
  - 密码/绑定信息变更（后续版本）：强制历史 token 失效
- 存储与传输：
  - token 只走 HTTPS
  - 前端本地存储需区分 `access` 与 `refresh`
  - 日志中禁止打印完整 token（最多显示前后各几位）
- 兼容策略：MVP 初期可先实现“重新登录换 token”，后续再补完整刷新机制

#### 2.5.3 元信息抓取边界（MVP）

- 抓取定位：辅助增强，不是保存前置条件
- 保存优先级：先落库原始链接，再异步抓取元信息
- 抓取超时：单次请求建议 `2-3` 秒，最多重试 `1` 次
- 抓取范围（仅轻量）：
  - 页面 `title`
  - 站点域名/来源平台
  - 可直接读取的 `og:title` / `og:image` / `author`（可选）
- 明确不做：
  - 登录态页面解析
  - 反爬对抗与复杂 JS 渲染
  - 正文抽取与 AI 深度理解（MVP 外）
- 失败处理：
  - 标记 `meta_status=FAILED`（或等价状态）
  - 不影响主链路“保存成功”
  - 允许后续手动重试补抓

### 2.6 日志规范（重点）

- 目标：只保留对人工审阅和排查真正有帮助的日志
- 开发环境保留 MyBatis SQL 控制台输出（用于看 SQL 与执行顺序）：
  - `mybatis.configuration.log-impl=org.apache.ibatis.logging.stdout.StdOutImpl`
  - `logging.level.<项目包名>.mapper=debug`
- Controller 只打印一条摘要日志：接口名 + 关键入参摘要 + 耗时 + 结果
- Service 只在关键分支和异常处打印日志，不打印流水账
- 禁止输出敏感信息：密码、验证码、token、手机号全量
- 非开发环境默认关闭 SQL 明细打印

### 2.7 Git 与提交规范

分支建议（个人项目简化版）：

- `master`：主分支，默认日常开发分支
- `feature/*`：重大功能或高风险改动时临时使用（可选）
- `fix/*`：需要隔离验证的修复任务时使用（可选）
- 完成后合并回 `master`，不强制保留长期 `develop` 分支

提交信息建议采用 Conventional Commits：

- `feat:` 新功能
- `fix:` 修复问题
- `refactor:` 重构
- `docs:` 文档更新
- `chore:` 工程维护

### 2.8 AI 协同开发约束

- AI 生成代码必须遵循本文件规范，不允许绕过目录与命名约定
- 关键业务逻辑（去重、标签合并、搜索条件）生成后必须人工复核
- 每次 AI 产出至少包含：变更说明、影响范围、验证方式

### 2.9 注释规范（便于人工审阅）

- 目标：注释用于解释“为什么这样做”，不是重复“代码做了什么”
- 仅在以下场景写注释：
  - 业务规则容易误解（如标签合并、去重、搜索组合条件）
  - 边界条件或兜底逻辑（空值、异常分支、兼容逻辑）
  - 非直观实现（性能权衡、历史约束、临时兼容）
- 禁止无效注释：
  - 与代码同义改写（如“给变量赋值”）
  - 过期注释不更新
  - 大段注释掉的旧代码（改用 Git 历史）
- 推荐做法：
  - 方法注释写输入/输出/副作用
  - 关键 SQL 或复杂条件前写一行“意图注释”
  - 注释保持短句、可读、可维护

## 3. 环境基线

### 3.1 开发环境版本基线

- OS：Windows + WSL2 Ubuntu（当前主要开发环境）
- JDK：21
- Maven：3.9+
- Node.js：22 LTS
- npm：10+
- Docker / Docker Compose：最新版稳定版
- MySQL：8.0
- IDE：IntelliJ IDEA（后端）+ HBuilderX（UniApp 打包）+ VS Code（文档与辅助编辑）

### 3.2 本地配置原则

- 本地开发默认使用 Docker MySQL，减少环境差异
- 配置项通过环境变量或 `.env` 管理，禁止硬编码账号密码
- 提供 `.env.example`，用于新成员与 AI 初始化环境

建议关键变量：

- `DB_HOST`
- `DB_PORT`
- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`
- `JWT_SECRET`
- `SMS_PROVIDER`（预留）

### 3.3 本地启动基线流程

1. 启动基础依赖（MySQL）
2. 执行数据库初始化脚本
3. 启动后端服务并完成健康检查
4. 启动 UniApp 开发服务（H5 调试）
5. Android 侧验证系统分享链路（真机优先）

### 3.4 Android 调试与部署基线

- Android MVP 以 APK 安装验证为主，不以上架流程为本轮目标
- 必测链路：外部 App 分享 URL -> LinkVault 承接页 -> 保存成功反馈
- 对于分享接入失败场景，需要有手动粘贴导入兜底
- 原生桥接代码保持最小化，避免与业务逻辑深耦合

### 3.5 Android 分享桥接约定（避免踩坑）

- 入口兼容：同时处理 `ACTION_SEND`（单条）与 `ACTION_SEND_MULTIPLE`（多条，MVP 可先取首条）
- 解析对象：优先读取 `EXTRA_TEXT`，兼容“标题 + 链接”混合文本
- 链接提取规则：
  - 从分享文本中提取第一个合法 URL 作为 `raw_url`
  - 同时保留原始分享文本 `raw_share_text`（便于排查）
  - 记录来源应用包名 `source_app`（如微信、B站、X）
- 与 `docs/linkdata/linkdata示例.md` 对齐的约束：
  - 微信内常见“预览文案 + 原始链接”混合内容，不能只按整段文本入库
  - 短链（如 `b23.tv`）允许直接保存，后端可异步展开
- 跳转约定：
  - 原生桥接只负责把解析结果传给导入页，不承载业务逻辑
  - URI 传参需做编码和长度保护，超长内容改为本地临时缓存 + ID 传递
- 兜底策略：
  - 提取失败时，导入页展示“可编辑原文”，允许用户手动修正链接再保存

## 4. 质量与验收门槛（MVP 阶段）

### 4.1 功能完成定义（DoD）

单个功能合并前至少满足：

- 对齐需求文档范围（不引入超范围能力）
- 核心路径可本地复现
- 关键接口有基础自测
- 代码符合命名、结构、异常处理规范
- 变更附最小文档说明

### 4.2 测试基线

后端测试为主，前端测试从轻：

后端测试框架说明：

- `JUnit 5`：基础单元测试框架（默认标准）
- `Spring Boot Test`：`Controller/Service` 集成测试与上下文启动
- `Mockito`：依赖隔离与行为模拟（单元测试中使用）
- `AssertJ`：可读性更高的断言
- `Testcontainers + MySQL`：数据库集成测试（尽量贴近真实 MySQL 行为）

后端测试分层与框架搭配：

- `Service` 单元测试：`JUnit 5 + Mockito + AssertJ`
- `Mapper`/数据层集成测试：`Spring Boot Test + Testcontainers(MySQL)`
- `Controller` 接口测试：`Spring Boot Test + MockMvc`

执行基线：

- 本地快速测试：`mvn test`
- 指定模块测试：`mvn -pl app-server test`
- CI 最低要求：每次提交至少跑通 `Service` 单元测试与 `Controller` 冒烟测试

- `Service` 单元测试（核心）
  - 必测：链接去重、标签合并、标签删除分支、搜索条件组合（关键词 + 多标签 AND）
  - 要求：每个核心业务规则至少 1 条正常用例 + 1 条边界/异常用例
- `Mapper` / 数据层集成测试（核心）
  - 必测：多标签交集查询、按用户隔离查询、去重唯一约束
  - 要求：基于真实 MySQL 结构执行，避免仅靠内存数据库
- `Controller` 接口测试（核心）
  - 必测：登录鉴权、导入保存、标签管理、搜索、详情编辑
  - 要求：覆盖状态码、返回体结构（`code/msg/data`）、鉴权失败分支
- 回归测试清单（发布前）
  - 登录 -> 导入保存 -> 备注 -> 打标签 -> 搜索 -> 详情编辑主链路全通过
  - 标签“重命名/合并/删除”时不破坏链接主记录

最低门槛（个人项目可执行版）：

- 新增或修改后端业务代码时，至少补 1 条自动化测试
- 核心模块（导入、标签、搜索）必须有可重复执行的测试用例
- 不再以“仅手工前后端联调”作为唯一验收依据

### 4.3 前后端联调（Playwright CLI + Playwright skill）

- 联调工具：使用已安装的 `playwright cli` + `playwright skill`
- 联调定位：用于端到端流程回归，不替代后端单元/集成测试
- 每次联调至少覆盖以下流程：
  - 登录 -> 新增链接 -> 编辑备注 -> 打多个标签 -> 搜索命中 -> 进入详情
- 联调结果要求：
  - 失败时记录失败步骤、页面截图/日志、对应后端接口
  - 修复后必须重跑失败用例并通过

### 4.4 不通过项

出现以下任一情况，不应进入 MVP 验收：

- 分享导入链路不稳定或经常丢链接
- 元信息抓取失败导致保存失败
- 标签删除/合并引起链接主记录异常丢失
- 搜索无法稳定覆盖“标题 + 备注 + 多标签交集”
- 仅靠手工联调且无核心自动化测试兜底

## 5. 阶段性实施建议

按以下顺序推进落地：

1. 固化仓库目录与 `.env.example`、`docker compose` 基础设施
2. 优先打通“登录 -> 导入保存 -> 备注 -> 标签 -> 搜索 -> 详情”主链路
3. 再补标签管理细则（置顶、重命名、合并、删除时序）
4. 最后补齐文档、测试与验收清单
