# LinkVault MVP 技术栈规范与环境基线

## 0. 文档目标与适用范围

本文档用于约束 LinkVault MVP 阶段的技术栈、开发规范、运行环境与验收边界。

适用范围：

- LinkVault Android MVP 阶段
- 后端、前端（UniApp / 后续 Web）、数据库与 SQL 开发
- AI 主导编码场景下的统一工程约束

约束原则：

- 先保证“保存-备注-标签-回找”主闭环可用，再做扩展能力
- 规范以“可执行、可复现、可验收”为目标，不追求过度设计
- 若本文档与需求、接口、数据库字段设计冲突，以对应专项文档为准，再回填修订本文档

## 1. 技术栈定稿

### 1.1 总体架构

采用“前后端分离 + UniApp 优先落地 + Web 后续补齐”的实现方式：

- 客户端：UniApp（Vue3）承担业务页面与交互
- Web 端：后续使用 Vue3 + Element Plus 复用同一套后端接口
- 后端：Spring Boot 提供 REST API
- 数据层：MyBatis-Plus + MySQL
- Android 端：不引入原生分享桥接，MVP 仅实现 APP 内手动粘贴 URL 导入
- H5 当前主要用于开发调试；正式 Web 前端在移动端主链路稳定后再启动

### 1.2 技术组件清单

后端：

- Java 21（LTS）
- Spring Boot 3.x
- MyBatis-Plus 3.5.x
- MySQL 8.0
- Maven 3.9+
- Lombok（简化实体类样板代码）
- jjwt 0.12.x（JWT 生成与解析）

前端（当前 MVP）：

- Node.js 24（当前 WSL 本地开发基线）
- UniApp（Vue3）
- uView Plus（移动端 UI 组件库，作为主要基础组件来源）
- Pinia（前端全局状态管理，仅用于登录态、当前用户等必要状态）
- HBuilderX（用于 Android 打包）

前端（后续 Web）：

- Vue3
- Element Plus（PC / Web 端 UI 组件库）

开发与运行环境：

- Docker + Docker Compose（作为后续部署方案预留，当前不作为本地开发前置条件）
- Git（版本管理）

MVP 明确不引入：

- Redis、消息队列、搜索引擎、对象存储
- refresh token、服务端 token 黑名单
- 账号密码登录、邮箱绑定、头像上传
- AI 总结、语义搜索、推荐系统
- Android 原生分享桥接、iOS、H5 正式产品化
- Figma 设计稿驱动开发流程

### 1.3 选型说明（MVP 视角）

- UniApp 满足“先完成 Android MVP 主链路”的目标，降低首版成本
- Spring Boot + MyBatis-Plus 能快速覆盖账号、收藏、标签、搜索等 CRUD 与关系查询场景
- MySQL 足够支撑当前班级项目规模与 MVP 数据量
- 去除原生分享桥接，优先把手动粘贴导入主链路做稳定
- 前端视觉暂不单独引入设计稿体系，先依赖 uView Plus / Element Plus 默认组件能力落地页面结构和交互

## 2. 开发规范基线

### 2.1 仓库结构约定

个人 Demo 项目优先采用简单结构，够用即可：

```text
linkvault/
├── backend/                        # Spring Boot 后端
├── frontend/                       # 当前 UniApp 前端（后续可扩展 web 子目录）
├── docs/                           # 项目文档与 SQL
└── docker/                         # 部署阶段容器编排与运行配置
```

说明：

- 后端默认单体单模块，避免一开始拆 `common/pojo/app-server` 多模块
- 只有在单模块明显变重时，再考虑按业务或公共能力拆分
- 文档和 SQL 可先放在 `docs/` 下，不强制过早细分过多子目录

### 2.2 后端规范（Java）

- 包结构按常规分层：`controller` / `service` / `service/impl` / `mapper` / `entity` / `dto` / `vo` / `exception` / `config`
- 后端默认单模块单体应用，优先降低工程复杂度
- 统一返回结构：`code`、`msg`、`data`
- HTTP 状态码与业务错误码必须按 `docs/engineering/接口文档.md` 维护，前端不得通过匹配 `msg` 做核心业务分支
- 参数校验统一使用 `jakarta.validation`
- 全局异常统一处理，禁止直接向前端暴露堆栈
- 数据库操作优先使用 MyBatis-Plus，复杂查询再补充 XML
- 命名统一小驼峰，布尔字段以 `is/has` 语义命名
- 分页结果统一分页对象（如 `PageResult<T>`），避免在各接口重复定义分页字段
- Entity 映射数据库表，DTO 表示请求入参，VO 表示接口出参；禁止直接把 Entity 返回给前端
- VO 字段名必须对齐接口文档，例如 `bookmarkId`、`originalUrl`、`publishedAt`、`savedAt`、`pinned`
- 查询当前用户数据时必须显式带 `user_id` 条件，不能只凭 `bookmarkId`、`tagId` 查询后再信任结果

后端核心业务边界：

- 登录注册统一走手机号 + 验证码，未注册手机号自动创建账号
- 新账号必须生成默认昵称，并通过 `isNewUser=true` 引导前端进入信息完善页
- 用户头像只允许 9 张系统预置静态资源路径，后端更新资料时必须做白名单校验
- 备注未填写时按空字符串处理，不使用 `null` 表达“无备注”
- 收藏删除采用物理删除当前用户的 `lv_bookmark` 与 `lv_bookmark_tag`，不删除 `lv_link`
- 标签删除只允许删除无关联标签；有关联标签必须先走合并流程

### 2.3 前端规范（UniApp/Vue3，当前 MVP）

- 前端代码风格统一使用 Vue3 组合式 API（Composition API）
- 页面与组件默认使用 `<script setup>` 写法
- 新增代码禁止使用 Options API（`data` / `methods` / `computed` 组织方式）
- 页面路由按业务域划分：登录、首页、导入承接、标签管理、搜索、详情、我的
- 组件命名：`PascalCase`，页面目录使用 `kebab-case`
- UI 基础组件优先使用 uView Plus，避免同时混用多套移动端组件库
- 视觉实现优先复用 uView Plus 默认组件样式，不为 MVP 额外设计主题、品牌色和高保真视觉稿
- 原型图对页面结构、字段、交互路径是强约束；对配色、图标、字体和精细视觉不是当前阶段重点
- API 调用统一封装（请求拦截、登录态注入、错误提示统一处理）
- 状态管理使用 Pinia；仅将登录态、当前用户等跨页面必要状态放入 store，列表筛选、表单输入、标签选择等页面态优先保留在页面内部
- 图标优先使用 uView Plus 内置图标；缺口再补充 uni-icons，禁止沿用原型里的 emoji 作为正式 UI 图标
- 底部一级页面固定为：首页、搜索、标签、我的
- 二级页面不展示底部导航，包括手动导入、标签筛选、链接详情、编辑用户资料
- 首页与“我的”页面按接口文档使用小接口组合，不新增 `/home/overview` 聚合接口
- 时间展示统一由前端把接口 UTC 时间转为 `Asia/Shanghai` 后展示，接口不返回相对时间文案
- 登录态失效时统一清理本地 token 并回到登录页
- 前端只保存必要登录态，不保存验证码明文

#### 2.3.1 前端实现优先级

当前阶段前端的首要目标不是视觉精修，而是把原型中的页面、字段、流程和交互稳定落地为可维护代码。

优先级顺序：

1. 页面结构与跳转对齐原型
2. 字段展示与接口文档对齐
3. 代码目录清晰、模块边界明确、便于 AI 逐页生成和人工维护
4. 优先复用 UI 库默认组件能力，不做大量自定义样式覆盖
5. 在主链路稳定后，再考虑统一主题、细化视觉和补 Web 端体验

MVP 阶段明确约束：

- 不要求先产出 Figma 或高保真设计稿
- 不要求先定义完整设计令牌体系
- 不为“审美统一”额外阻塞页面开发
- 若默认组件已能表达原型结构和交互，则优先直接落地代码

最低可用视觉与交互要求：

- 保持页面结构清晰，列表、卡片、表单分区明确
- 使用同一套 UI 库组件，不混搭风格明显不同的第三方组件
- 删除、退出、覆盖等破坏性操作必须有确认反馈
- 表单、列表、弹窗优先保证可读性和可操作性，而不是追求视觉装饰
- 主要可点击区域最小触控尺寸为 `44px * 44px`
- 输入框必须有明确 label、placeholder 或等价上下文
- 颜色不能作为唯一状态表达，选中态需同时提供图标、边框、文字或结构变化

### 2.4 数据库与 SQL 规范

- 表名使用 `lv_` 前缀，核心表与数据库字段设计保持一致：
  - `lv_user`：用户账号
  - `lv_sms_code`：短信验证码记录
  - `lv_link`：链接对象（原始 URL、元信息，可被多个用户共享）
  - `lv_bookmark`：用户收藏关系（用户 + 链接 + 当前用户原始 URL + 备注 + 保存时间，私有数据挂这里）
  - `lv_tag`：标签（归属某一用户）
  - `lv_bookmark_tag`：收藏与标签的多对多关系
- 必备审计字段：`created_at`、`updated_at`
- 时间字段统一使用 `DATETIME(3)`，按 UTC 写入和读取
- 所有外键关系在应用层与索引层保证一致性（MVP 可不强制物理外键）
- 索引优先覆盖：用户维度查询、链接去重、标签关系、搜索关键路径
- MVP 阶段收藏和标签删除采用物理删除，避免软删除状态与唯一索引产生冲突；`lv_link` 不因用户删除收藏而删除
- SQL 脚本统一放在 `docs/sql`，并保持可重复初始化
- `lv_link.original_url` 仅用于排查；接口展示和打开原始链接必须使用 `lv_bookmark.original_url`
- `lv_bookmark.note` 未填写时保存为空字符串，接口返回 `""`
- 平台枚举使用 `BILIBILI`、`YOUTUBE`、`X`、`WECHAT_OFFICIAL_ACCOUNT`、`WEB`、`UNKNOWN`
- 标签置顶字段为 `is_pinned`，接口 VO 对外映射为 `pinned`

### 2.5 API 约定

- 接口前缀统一：`/api/v1`
- 鉴权方式：登录后下发 JWT，后续请求通过请求头携带
- 请求头规范：统一使用 `Authorization: Bearer <token>`
- 分页参数统一：`page`、`pageSize`
- 接口时间字段统一为 UTC ISO 8601 字符串，带 `Z` 后缀；数据库 `DATETIME(3)` 按 UTC 写入和读取
- 错误码分层：通用错误、鉴权错误、业务错误
- 成功响应统一 `code=0`、`msg=success`
- 失败响应统一非 `0` 业务错误码，错误码扩展必须同步更新接口文档
- 列表类 VO 复用 `BookmarkPageVO`；标签列表复用 `TagListVO`
- 首页最近保存使用 `GET /api/v1/bookmarks?page=1&pageSize=5`
- 全部收藏、标签筛选、未打标签、关键词搜索统一使用 `GET /api/v1/bookmarks/search`
- “我的”页面只调用 `GET /api/v1/me` 和 `GET /api/v1/me/stats`

#### 2.5.1 项目时间策略（中国用户 + 国外服务器）

- 项目主要面向中国用户，但后端可能部署在美国或其他非中国时区服务器；所有业务时间不得依赖服务器本地时区
- 后端内部表示真实时刻时优先使用 `Instant`；生成当前时间时使用 `Instant.now()`，避免直接使用受 JVM 默认时区影响的 `LocalDateTime.now()`
- 数据库继续使用 `DATETIME(3)`，但所有写入值必须先转换为 UTC；读取时也必须按 UTC 解释
- 接口统一输出 UTC ISO 8601 毫秒格式，带 `Z` 后缀，例如 `2026-04-16T08:12:36.000Z`
- 元信息解析采用“宽进严出”：解析层兼容来源时间有无毫秒、时间戳、带时区字符串和平台本地时间；进入业务对象和接口前统一归一为 UTC
- 面向普通用户展示的时间统一转换为北京时间 `Asia/Shanghai`，例如 `2026-04-16T08:12:36.000Z` 展示为 `2026-04-16 16:12`
- 排序、筛选、过期判断、验证码有效期判断等计算逻辑统一基于 UTC instant 或数据库 UTC 值，不基于北京时间展示字符串
- 若普通网页或平台页面给出的发布时间没有时区且无法可靠推断，允许 `published_at` 为空，不写入误导性时间

#### 2.5.2 短信验证码方案（MVP）

- 目标：先保证可登录、可联调
- 验证码有效期：`5` 分钟
- 发送频率限制：同手机号 `60` 秒内只能发送 `1` 次
- 校验失败限制：同一验证码最多失败 `5` 次，超过后失效
- 使用限制：验证码登录成功后立即标记已使用，不能重复使用
- 存储要求：验证码仅存服务端（数据库临时表），禁止前端缓存明文
- 环境约定：
  - `dev`：允许 Mock 短信通道（固定验证码或控制台输出）
  - `prod`：接入真实短信服务时再补更完整风控
- 接口返回：不暴露“号码是否已注册”，统一返回文案

#### 2.5.3 JWT 生命周期约定（MVP）

- `access token`：有效期 `7` 天，过期后重新登录
- 失效策略：用户主动退出登录时，前端清除本地 token；MVP 不实现服务端 token 黑名单或 refresh token 机制
- 存储与传输：
  - 生产或外网联调环境 token 只走 HTTPS
  - 前端存储于 `uni.setStorageSync`
  - 日志中禁止打印完整 token
- `JWT_SECRET` 不得提交到仓库；本地使用 `.env` 或 IDE 运行配置注入
- token 载荷只放必要身份信息，例如 `userId`、签发时间和过期时间，不放手机号全量、验证码或隐私数据

#### 2.5.4 元信息抓取边界（MVP）

- 抓取定位：辅助增强，不是保存前置条件
- 保存优先级：先落库当前用户粘贴的原始链接，再异步抓取元信息
- 异步实现：使用 Spring `@Async` + 独立线程池，不引入消息队列
- 抓取超时：单次请求建议 `2-3` 秒，最多重试 `1` 次
- 元信息展示优先级：
  - 核心展示字段：`title`、`platform`、`publisher`
  - 次要展示字段：`published_at`
- 抓取范围（仅轻量）：
  - 页面 `title`
  - 站点域名/来源平台
  - 可直接读取的 `author` / `publisher` / `og:title` / `article:published_time` / `publish_time`
- 提取与兜底规则：
  - `platform` 必须落库，优先根据最终域名、短链跳转结果和平台规则识别
  - `title` 必须作为展示字段存在；解析失败时使用兜底文案 `未解析标题`
  - `publisher` 必须作为展示字段存在；解析失败时使用兜底文案 `未知作者`
  - `published_at` 尽量提取；提取不到允许为空，不影响保存与展示主流程
- 明确不做：
  - 登录态页面解析
  - 反爬对抗与复杂 JS 渲染
  - 正文抽取与 AI 深度理解（MVP 外）
- 失败处理：
  - 核心字段全部成功提取时标记 `meta_status=SUCCESS`
  - 仅依赖部分真实字段或兜底文案完成展示时标记 `meta_status=PARTIAL`
  - 关键解析流程整体失败时标记 `meta_status=FAILED`
  - 不影响主链路“保存成功”
  - MVP 不提供手动重试元信息抓取接口；后续若补能力，必须先更新接口文档和原型入口

### 2.6 日志规范（重点）

- 目标：只保留对人工审阅和排查真正有帮助的日志
- 开发环境保留 MyBatis SQL 控制台输出（用于看 SQL 与执行顺序）：
  - `mybatis.configuration.log-impl=org.apache.ibatis.logging.stdout.StdOutImpl`
  - `logging.level.<项目包名>.mapper=debug`
- Controller 只打印一条摘要日志：接口名 + 关键入参摘要 + 耗时 + 结果
- Service 只在关键分支和异常处打印日志，不打印流水账
- 禁止输出敏感信息：密码、验证码、token、手机号全量
- 手机号日志只允许脱敏展示，例如 `138****0000`
- URL 日志默认只打印域名、平台和 `bookmarkId`，避免长 URL 与 token 参数污染日志
- 非开发环境默认关闭 SQL 明细打印

### 2.7 Git 与提交规范

分支建议（个人项目简化版）：

- `master`：主分支，默认日常开发分支
- `feature/*`：重大功能或高风险改动时临时使用（可选）
- `fix/*`：需要隔离验证的修复任务时使用（可选）
- 完成后合并回 `master`，不强制保留长期 `develop` 分支

提交信息建议采用 Conventional Commits：

- `feat:` 新功能
- `fix:` 修复问题
- `refactor:` 重构
- `docs:` 文档更新
- `chore:` 工程维护

### 2.8 AI 协同开发约束

- AI 生成代码必须遵循本文件规范，不允许绕过目录与命名约定
- 本项目默认由 AI 负责实际编码，用户主要负责需求定义与环境配置
- AI 完成代码后必须先自行运行可执行的本地验证，如启动服务、调用接口、检查日志、走通页面主流程等
- 每次 AI 完成改动后，至少给出：变更说明、影响范围、AI 已执行的验证方式、建议用户手工复核点

### 2.9 注释规范（便于人工审阅）

- 目标：注释用于解释“为什么这样做”，不是重复“代码做了什么”
- 仅在以下场景写注释：
  - 业务规则容易误解（如标签合并、去重、搜索组合条件）
  - 边界条件或兜底逻辑（空值、异常分支、兼容逻辑）
  - 非直观实现（性能权衡、历史约束、临时兼容）
- 禁止无效注释：
  - 与代码同义改写（如“给变量赋值”）
  - 过期注释不更新
  - 大段注释掉的旧代码（改用 Git 历史）
- 推荐做法：
  - 方法注释写输入/输出/副作用
  - 关键 SQL 或复杂条件前写一行“意图注释”
  - 注释保持短句、可读、可维护

## 3. 环境基线

### 3.1 开发环境版本基线

- OS：Windows + WSL2 Ubuntu（当前主要开发环境）
- JDK：21
- Maven：3.9+
- Node.js：24（以当前 WSL 实际开发环境为准）
- npm：10+
- Docker / Docker Compose：最新版稳定版
- MySQL：8.0
- IDE：IntelliJ IDEA（后端）+ HBuilderX（UniApp 打包）+ VS Code（文档与辅助编辑）

### 3.2 本地配置原则

- 本地开发默认采用“应用本机直接运行 + 复用已有本地依赖”的模式，优先保证调试效率并减少环境折腾
- 后端与 UniApp 开发服务优先在 WSL 本机启动，便于断点、热更新与日志排查
- WSL 环境中如已具备稳定可用的 MySQL，开发阶段优先直接复用本地实例，不额外引入 Docker MySQL
- 仅在进入部署验证、环境复刻或团队统一交付时，再补 Docker / Docker Compose 配置
- 配置项通过环境变量或 `.env` 管理，禁止硬编码账号密码
- 提供 `.env.example`，便于 AI 和后续自己快速复现环境
- 前端 `package.json` 建议补充 Node 版本声明（如 `engines.node`），避免团队环境与锁文件漂移

建议关键变量：

- `DB_HOST`
- `DB_PORT`
- `DB_NAME`
- `DB_USER`
- `DB_PASSWORD`
- `JWT_SECRET`
- `JWT_EXPIRES_SECONDS`
- `SMS_PROVIDER`（预留）
- `APP_CORS_ALLOWED_ORIGINS`
- `LINK_META_TIMEOUT_MS`

### 3.3 本地启动基线流程

1. 确认本地 MySQL 已启动并可连接
2. 执行 `docs/sql/001_schema.sql`
3. 按需执行 `docs/sql/002_seed_test_data.sql`
4. 在 WSL 本机启动后端服务并完成健康检查
5. 在 WSL 本机启动 UniApp 开发服务（H5 调试）
6. 完成本地主链路联调：登录、保存、备注、标签、搜索、详情回查
7. Android 侧验证 APP 内手动粘贴导入链路（真机优先）
8. 本地开发与验收通过后，再进入 Docker 化部署验证

### 3.4 Android 调试与部署基线

- Android MVP 以 APK 安装验证为主，不以上架流程为本轮目标
- 必测链路：用户复制 URL -> 打开 LinkVault -> 粘贴到输入区 -> 保存成功反馈
- 不再纳入 Android 系统分享接入与原生桥接开发
- Android 客户端产物为 APK/安装包，不属于 Docker 容器部署对象

### 3.5 Docker 部署基线

- Docker 化目标：在本地功能联调通过后，再用容器方式复现接近部署环境的运行结果，而不是阻塞前期开发
- Docker 部署范围优先限定为服务端组件：`backend` + `mysql`（如有需要可后续补 `nginx` 反向代理）
- UniApp Android 客户端仍按本地构建 APK 的方式交付，不通过 Docker 发布到手机
- Docker Compose 配置需支持环境变量注入，避免将数据库账号、JWT 密钥等写死在镜像或仓库中
- 若本地开发阶段已使用宿主机 MySQL，部署时需补齐数据库初始化、账号权限、字符集与时区配置，避免“本地可用、容器不可用”；MySQL、JVM 与容器运行环境建议显式固定 UTC，用户侧展示再转北京时间
- 部署验收最低要求：
  - 容器可一键启动
  - 后端健康检查通过
  - 数据库连接正常
  - 核心 API 可用
  - 与本地联调通过的主闭环行为一致

### 3.6 手动粘贴导入约定（MVP）

- 输入对象：用户手动粘贴到输入框中的单个 `http://` 或 `https://` URL；若粘贴内容包含分享文案，本轮只允许其中出现一个 URL，并由前端提取该 URL 后提交
- 数据归属：前端提交给后端的 URL 保存到 `lv_bookmark.original_url`，链接对象 `lv_link` 只保存可复用元信息和首次入库原始 URL 作为排查字段
- 前端校验：
  - 去除首尾空白字符
  - 只接受 `http://` 或 `https://` URL
  - 粘贴文本中出现多个 URL 时直接提示用户只保留一个链接
  - 提交给后端的 `url` 字段必须是提取后的单个 URL，不提交整段分享文案
- 失败提示：
  - 非法 URL 直接前端提示，不发起保存请求
  - 超长 URL 或空字符串直接拦截
- 兜底策略：
  - 元信息抓取失败不影响保存
  - 承接页、列表页、详情页都应保留 `平台 / 作者 / 标题` 三个展示位，不因缺失而隐藏字段
  - 允许用户保存后再补备注与标签

## 4. 质量与验收门槛（MVP 阶段）

### 4.1 功能完成定义（DoD）

单个功能合并前至少满足：

- 对齐需求文档范围（不引入超范围能力）
- 对齐接口文档和数据库字段设计，不私自新增接口字段或表字段
- 核心路径可本地复现
- AI 已完成必要的本地运行验证，并说明验证结果
- 代码符合命名、结构、异常处理规范
- 变更附最小文档说明

### 4.2 本地运行验证基线

本项目 MVP 阶段默认验收方式为：

- AI 完成代码后，先在本地启动相关服务并执行必要的运行验证
- AI 需要尽量通过接口调用、页面操作、日志检查、数据库结果检查等方式确认改动可用
- 用户在 AI 验证通过后，再按主链路进行人工测试和最终确认
- 验证中发现的问题应先修复并重新验证，再交给用户手工复核

后端运行验证建议：

- 服务能正常启动，健康检查或启动日志无关键异常
- 新增或修改接口能通过实际请求验证主要成功路径和关键失败路径
- 涉及数据库写入时，应检查关键表数据是否符合预期
- 涉及鉴权、用户隔离、标签合并、删除等高风险逻辑时，应额外验证越权、重复操作、空数据等边界

前端运行验证建议：

- H5 调试环境能正常启动，页面无明显编译错误和运行时异常
- 新增或修改页面能完成主要点击、输入、跳转和接口交互
- 移动端关键页面应尽量通过浏览器移动视口或真机调试检查布局
- Android APK 构建和真机验证在进入交付前完成

### 4.3 人工联调

- 联调定位：用于验证主流程真实可用，是 MVP 阶段的主要验收方式
- 至少覆盖以下主链路：
  - 登录 -> 首次信息完善 -> 手动导入 -> 编辑备注 -> 打多个标签 -> 搜索命中 -> 进入详情
  - 首页统计/最近保存 -> 全部收藏 -> 未打标签筛选
  - 标签管理 -> 重命名 -> 合并 -> 删除无关联标签
- AI 完成自验后，应提供建议人工复核清单，用户按清单和自身使用习惯手工确认

### 4.4 不通过项

出现以下任一情况，不应进入 MVP 验收：

- 手动粘贴导入链路不稳定或经常丢链接
- 元信息抓取失败导致保存失败
- 标签删除/合并引起链接主记录异常丢失
- 搜索无法稳定覆盖“标题 + 备注 + 多标签交集”
- AI 未完成基础运行验证，或用户人工测试发现主链路不可用

## 5. 阶段性实施建议

按以下顺序推进落地：

1. 先固化最小可运行骨架：后端、前端、本地 MySQL、本地配置
