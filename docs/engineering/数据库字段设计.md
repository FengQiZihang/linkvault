# LinkVault MVP 数据库字段设计

## 0. 文档目的

本文用于定义 LinkVault MVP 阶段的 MySQL 数据库字段基线，服务后续后端实体、Mapper、接口与初始化 SQL 编写。

设计目标：

- 支撑“保存、备注、打标签、回找”主闭环
- 明确链接元信息与用户私有收藏数据的边界
- 落实 URL 解析实验中已验证的 `platform / publisher / title / published_at` 四个字段
- 避免过早引入复杂账号、全文检索、历史版本和分析统计能力

适用范围：

- MySQL 8.0
- Spring Boot + MyBatis-Plus
- MVP 单体后端

## 1. 总体设计原则

- 表名统一使用 `lv_` 前缀
- 主键统一使用 `BIGINT UNSIGNED`
- 时间字段统一使用 `DATETIME(3)`
- 字符集建议使用 `utf8mb4`
- 审计字段统一包含 `created_at`、`updated_at`
- MVP 阶段不强制使用物理外键，关系一致性由应用层和索引约束保证
- 链接对象与用户收藏关系分离，避免不同用户的备注、标签、保存动作互相污染

## 2. 核心表清单

| 表名 | 用途 | MVP 必需 |
| --- | --- | --- |
| `lv_user` | 用户账号 | 是 |
| `lv_sms_code` | 短信验证码临时记录 | 是 |
| `lv_link` | 链接对象与可复用元信息 | 是 |
| `lv_bookmark` | 用户收藏关系与私有备注 | 是 |
| `lv_tag` | 用户标签 | 是 |
| `lv_bookmark_tag` | 收藏与标签多对多关系 | 是 |

## 3. 字段约定

### 3.1 ID 与时间

| 字段 | 类型 | 说明 |
| --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 主键 |
| `created_at` | `DATETIME(3)` | 创建时间 |
| `updated_at` | `DATETIME(3)` | 更新时间 |

### 3.2 状态字段

状态字段优先使用短字符串，便于日志、接口和人工排查。

示例：

- `meta_status`: `PENDING` / `SUCCESS` / `PARTIAL` / `FAILED`
- `status`: `ACTIVE` / `DELETED`

## 4. 用户表：`lv_user`

用途：

- 承载账号基础信息
- 支撑手机号验证码登录
- 保证收藏数据按用户隔离

字段设计：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 是 | 无 | 用户 ID |
| `phone` | `VARCHAR(20)` | 是 | 无 | 手机号，登录唯一标识 |
| `nickname` | `VARCHAR(50)` | 是 | 无 | 昵称，首次注册后设置 |
| `avatar_url` | `VARCHAR(512)` | 否 | `NULL` | 头像地址，MVP 可为空或使用默认头像 |
| `status` | `VARCHAR(20)` | 是 | `ACTIVE` | 用户状态 |
| `last_login_at` | `DATETIME(3)` | 否 | `NULL` | 最近登录时间 |
| `created_at` | `DATETIME(3)` | 是 | 当前时间 | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | 当前时间 | 更新时间 |

索引建议：

| 索引名 | 字段 | 类型 | 说明 |
| --- | --- | --- | --- |
| `uk_user_phone` | `phone` | 唯一索引 | 保证手机号唯一 |

约束说明：

- MVP 不设计账号密码字段
- 邮箱为后续能力，当前不建字段
- 头像可先由前端默认图或后端默认值处理

## 5. 短信验证码表：`lv_sms_code`

用途：

- 保存验证码发送与校验记录
- 支撑开发环境 Mock 短信通道
- 避免验证码明文存在前端

字段设计：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 是 | 无 | 主键 |
| `phone` | `VARCHAR(20)` | 是 | 无 | 手机号 |
| `code` | `VARCHAR(10)` | 是 | 无 | 验证码，MVP 可先明文，后续可改 hash |
| `scene` | `VARCHAR(30)` | 是 | `LOGIN` | 使用场景，如登录/注册 |
| `expires_at` | `DATETIME(3)` | 是 | 无 | 过期时间 |
| `used_at` | `DATETIME(3)` | 否 | `NULL` | 使用时间 |
| `send_ip` | `VARCHAR(64)` | 否 | `NULL` | 发送请求 IP |
| `created_at` | `DATETIME(3)` | 是 | 当前时间 | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | 当前时间 | 更新时间 |

索引建议：

| 索引名 | 字段 | 类型 | 说明 |
| --- | --- | --- | --- |
| `idx_sms_phone_created` | `phone`, `created_at` | 普通索引 | 校验发送频率和取最新验证码 |
| `idx_sms_expires` | `expires_at` | 普通索引 | 清理过期验证码 |

约束说明：

- 验证码有效期为 `5` 分钟
- 同手机号 `60` 秒内只能发送 `1` 次
- MVP 可先保留明文验证码，正式生产前建议改为 hash 存储

## 6. 链接对象表：`lv_link`

用途：

- 承载链接本身和可复用元信息
- 不保存用户备注、标签等私有数据
- 多个用户保存同一链接时可共享同一条链接元信息

字段设计：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 是 | 无 | 链接 ID |
| `original_url` | `VARCHAR(2048)` | 是 | 无 | 用户粘贴的原始 URL |
| `normalized_url` | `VARCHAR(2048)` | 是 | 无 | 用于去重的规范化 URL |
| `final_url` | `VARCHAR(2048)` | 否 | `NULL` | 短链跳转后的最终 URL |
| `url_hash` | `CHAR(64)` | 是 | 无 | `normalized_url` 的 SHA-256，用于唯一索引 |
| `platform` | `VARCHAR(50)` | 是 | `UNKNOWN` | 信息源/发布平台，如 `BILIBILI`、`YOUTUBE`、`X`、`WECHAT_MP` |
| `publisher` | `VARCHAR(255)` | 是 | `未知作者` | 发布作者、UP 主、频道、公众号或账号名 |
| `title` | `VARCHAR(500)` | 是 | `未解析标题` | 标题；普通 X 帖子可使用正文摘要 |
| `published_at` | `DATETIME(3)` | 否 | `NULL` | 内容发布时间，可为空 |
| `meta_status` | `VARCHAR(20)` | 是 | `PENDING` | 元信息抓取状态 |
| `meta_fetched_at` | `DATETIME(3)` | 否 | `NULL` | 最近一次元信息抓取完成时间 |
| `meta_error` | `VARCHAR(1000)` | 否 | `NULL` | 最近一次抓取失败原因摘要 |
| `created_at` | `DATETIME(3)` | 是 | 当前时间 | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | 当前时间 | 更新时间 |

索引建议：

| 索引名 | 字段 | 类型 | 说明 |
| --- | --- | --- | --- |
| `uk_link_url_hash` | `url_hash` | 唯一索引 | 链接级去重 |
| `idx_link_platform` | `platform` | 普通索引 | 平台筛选与统计预留 |
| `idx_link_meta_status` | `meta_status`, `meta_fetched_at` | 普通索引 | 查找待抓取或失败重试记录 |
| `idx_link_title` | `title` | 普通索引 | 标题 LIKE 搜索的基础辅助索引 |

字段说明：

- `original_url` 保留用户最初粘贴内容，便于排查和展示原始来源
- `normalized_url` 用于去重，建议去除无意义追踪参数后生成
- `final_url` 用于记录短链展开结果，如 `b23.tv`、`youtu.be`
- `url_hash` 用于避免 `VARCHAR(2048)` 直接做唯一索引造成索引过长
- `published_at` 是内容发布时间，不等同于用户保存时间
- `created_at` 是链接对象首次入库时间，不等同于内容发布时间

平台枚举建议：

| 值 | 说明 |
| --- | --- |
| `BILIBILI` | Bilibili 视频 |
| `YOUTUBE` | YouTube 视频 |
| `X` | X / Twitter 文章或帖子 |
| `WECHAT_MP` | 微信公众号文章 |
| `WEB` | 普通网页 |
| `UNKNOWN` | 未识别平台 |

元信息状态建议：

| 值 | 说明 |
| --- | --- |
| `PENDING` | 已保存，等待抓取 |
| `SUCCESS` | 核心字段和可用发布时间均成功获取 |
| `PARTIAL` | 三个核心字段可展示，但部分字段使用兜底或发布时间缺失 |
| `FAILED` | 抓取失败，仅保留 URL 和兜底展示字段 |

## 7. 用户收藏表：`lv_bookmark`

用途：

- 表示某个用户保存了某个链接
- 保存用户私有备注、保存时间和删除状态
- 同一用户对同一链接只保留一条收藏记录

字段设计：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 是 | 无 | 收藏 ID |
| `user_id` | `BIGINT UNSIGNED` | 是 | 无 | 所属用户 |
| `link_id` | `BIGINT UNSIGNED` | 是 | 无 | 对应链接 |
| `note` | `VARCHAR(1000)` | 否 | `NULL` | 用户备注 |
| `status` | `VARCHAR(20)` | 是 | `ACTIVE` | 收藏状态 |
| `saved_at` | `DATETIME(3)` | 是 | 当前时间 | 用户保存时间 |
| `deleted_at` | `DATETIME(3)` | 否 | `NULL` | 删除时间，软删除使用 |
| `created_at` | `DATETIME(3)` | 是 | 当前时间 | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | 当前时间 | 更新时间 |

索引建议：

| 索引名 | 字段 | 类型 | 说明 |
| --- | --- | --- | --- |
| `uk_bookmark_user_link` | `user_id`, `link_id` | 唯一索引 | 同一用户同一链接只收藏一次 |
| `idx_bookmark_user_saved` | `user_id`, `saved_at` | 普通索引 | 首页最近保存列表 |
| `idx_bookmark_user_status` | `user_id`, `status`, `saved_at` | 普通索引 | 当前用户有效收藏查询 |

约束说明：

- `note` 属于用户私有数据，不放在 `lv_link`
- 删除收藏只删除当前用户与链接的关系，不删除 `lv_link`
- MVP 采用软删除更便于排查和恢复；如后续要物理删除，需要同步清理标签关系

## 8. 标签表：`lv_tag`

用途：

- 保存用户自定义标签
- 标签只归属于单个用户
- 支撑置顶层与普通层展示

字段设计：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 是 | 无 | 标签 ID |
| `user_id` | `BIGINT UNSIGNED` | 是 | 无 | 所属用户 |
| `name` | `VARCHAR(20)` | 是 | 无 | 标签名称 |
| `is_pinned` | `TINYINT(1)` | 是 | `0` | 是否置顶 |
| `status` | `VARCHAR(20)` | 是 | `ACTIVE` | 标签状态 |
| `created_at` | `DATETIME(3)` | 是 | 当前时间 | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | 当前时间 | 更新时间 |

索引建议：

| 索引名 | 字段 | 类型 | 说明 |
| --- | --- | --- | --- |
| `uk_tag_user_name` | `user_id`, `name` | 唯一索引 | 同一用户下标签名唯一 |
| `idx_tag_user_pinned` | `user_id`, `is_pinned`, `created_at` | 普通索引 | 标签两层展示 |
| `idx_tag_user_status` | `user_id`, `status` | 普通索引 | 查询有效标签 |

约束说明：

- 未打标签是状态，不创建“未打标签”实体标签
- 标签名称长度限制为 `20` 字以内
- 标签重命名若命中已有标签，应按合并流程处理
- 第一层置顶标签只记录 `is_pinned`，MVP 不提供手动排序字段

## 9. 收藏标签关系表：`lv_bookmark_tag`

用途：

- 保存收藏与标签的多对多关系
- 支撑多标签筛选和标签合并

字段设计：

| 字段 | 类型 | 必填 | 默认值 | 说明 |
| --- | --- | --- | --- | --- |
| `id` | `BIGINT UNSIGNED` | 是 | 无 | 主键 |
| `bookmark_id` | `BIGINT UNSIGNED` | 是 | 无 | 收藏 ID |
| `tag_id` | `BIGINT UNSIGNED` | 是 | 无 | 标签 ID |
| `user_id` | `BIGINT UNSIGNED` | 是 | 无 | 冗余用户 ID，便于按用户清理和查询 |
| `created_at` | `DATETIME(3)` | 是 | 当前时间 | 创建时间 |
| `updated_at` | `DATETIME(3)` | 是 | 当前时间 | 更新时间 |

索引建议：

| 索引名 | 字段 | 类型 | 说明 |
| --- | --- | --- | --- |
| `uk_bookmark_tag` | `bookmark_id`, `tag_id` | 唯一索引 | 防止同一收藏重复打同一标签 |
| `idx_bt_tag_bookmark` | `tag_id`, `bookmark_id` | 普通索引 | 标签筛选列表 |
| `idx_bt_user_tag` | `user_id`, `tag_id` | 普通索引 | 用户维度标签关系查询 |
| `idx_bt_user_bookmark` | `user_id`, `bookmark_id` | 普通索引 | 查询某收藏的标签 |

约束说明：

- `user_id` 是冗余字段，正式写入时必须与 `lv_bookmark.user_id` 和 `lv_tag.user_id` 一致
- 标签合并时需要处理 `uk_bookmark_tag` 冲突，目标标签关系已存在时直接丢弃源关系

## 10. 关键查询支撑

### 10.1 首页最近保存

核心条件：

- `lv_bookmark.user_id = 当前用户`
- `lv_bookmark.status = ACTIVE`
- 按 `saved_at DESC`

涉及表：

- `lv_bookmark`
- `lv_link`

展示字段：

- `bookmark_id`
- `link_id`
- `title`
- `platform`
- `publisher`
- `published_at`
- `note`
- `saved_at`

### 10.2 关键词搜索

MVP 搜索范围：

- `lv_link.title`
- `lv_bookmark.note`

辅助展示：

- `lv_link.platform`
- `lv_link.publisher`

说明：

- MVP 使用 `LIKE` 足够
- 后续数据规模增长后可再考虑全文索引或外部搜索引擎

### 10.3 多标签交集筛选

实现语义：

- 多标签为 `AND`
- 收藏必须同时拥有全部选中标签

建议实现：

- 从 `lv_bookmark_tag` 按 `tag_id IN (...)` 聚合
- `GROUP BY bookmark_id`
- `HAVING COUNT(DISTINCT tag_id) = 选中标签数量`

### 10.4 未打标签筛选

实现语义：

- 用户收藏存在
- 该收藏没有任何标签关系

建议实现：

- `lv_bookmark LEFT JOIN lv_bookmark_tag`
- 条件 `lv_bookmark_tag.id IS NULL`

## 11. 去重策略

MVP 去重范围：

- 链接对象层：通过 `lv_link.url_hash` 去重
- 用户收藏层：通过 `lv_bookmark(user_id, link_id)` 去重

推荐流程：

1. 接收用户粘贴的 `original_url`
2. 生成 `normalized_url`
3. 计算 `url_hash`
4. 查询或创建 `lv_link`
5. 查询或创建 `lv_bookmark`
6. 异步抓取或补齐链接元信息

说明：

- 不建议直接以 `original_url` 做唯一键
- 短链展开后的 `final_url` 可辅助去重，但 MVP 阶段不要让短链解析阻断保存
- 若后续发现同一内容存在多个短链形式，可补“链接合并”能力

## 13. 不纳入 MVP 的表

以下表当前不建：

- 用户密码表
- refresh token 表
- 邮箱绑定表
- 链接元信息历史表
- 抓取任务队列表
- 全文搜索索引表
- 用户行为统计表
- 标签操作审计表

原因：

- 当前阶段目标是完成主闭环
- 上述表会增加工程复杂度，但不会显著提升 MVP 验证价值

## 14. 后续生成 SQL 时的注意事项

- 所有表使用 `ENGINE=InnoDB DEFAULT CHARSET=utf8mb4`
- 所有 `created_at`、`updated_at` 建议由数据库默认值和后端填充双重保障
- MyBatis-Plus 实体应与字段默认值保持一致
- 初始化 SQL 应放在 `docs/sql`
- 第一版 SQL 应保证可重复初始化开发库
