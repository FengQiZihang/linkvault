# 项目背景与AI协作提示

## 项目基本介绍

当前项目是一个全程使用 AI 辅助创建、用于解决自己实际需求的 APP。

## 当前文档状态

目前，`docs` 目录下已经有了较为丰富的编码前准备文档，并且已经写好了一个单文件原型图，格式为 HTML。

有了这些文档之后，我曾直接让 AI 生成前后端项目代码，但生成的代码量过大，自己一时无法有效测试和修改。
因此，后续又通过讨论，将编码阶段进一步细分，并补充可能缺失的设计内容。

目前，前后端历史代码已经全部删除，项目重新回到“以文档和原型为基线、从清晰工程骨架重新开始”的状态。

## 原型与设计现状

原型图本身就是 Claude Code使用Skill（UI UX Pro Max）完成设计的，经过多轮调整，页面布局和交互流程都已经基本确定。
目前原型的主要价值是固定页面结构、字段展示和交互路径；配色和 UI 风格不再作为当前阶段的关键问题处理。

## 前端实现策略

当前前端策略调整为：

- 不先补 Figma 设计稿
- 不先追求高保真视觉和品牌化配色
- 先以原型图作为页面结构与交互规范，直接落地前端代码
- 移动端使用 `UniApp + Vue3 + uView Plus` 默认组件能力完成 MVP
- 后续 Web 端使用 `Vue3 + Element Plus` 复用同一套后端接口与页面语义

当前阶段前端的重点是：

- 页面结构还原准确
- 交互流程和接口字段一致
- 代码组织清晰、模块边界明确、便于 AI 持续增量生成
- 在功能完成后，再决定是否补主题定制、统一视觉或设计稿沉淀

基于这个判断，目前前端项目将以更清晰的工程规范重新开始，优先保证模块化和可维护性，而不是先做视觉收敛。

## 后端与测试策略

当前也重新考虑了最终测试问题。
如果直接进行前后端联调，测试效率偏低，需要同时关注的事项过多。

目前计划是：

1. 先基于一份合格的接口文档生成后端项目。
2. 使用 Apifox 先完成接口测试与后端跑通。
3. 在后端稳定后，再继续完成前端实现与联调。

## 给 AI 的协作提示

在后续协作中，应优先利用 `docs` 目录下现有文档作为项目背景输入，避免一次性生成过大规模代码。
实现时应倾向于分阶段推进，先拆清页面和模块边界，再逐步生成、验证和修正代码。
除非后续明确进入视觉统一阶段，否则不要默认要求补 Figma、主题系统或高保真设计稿。
