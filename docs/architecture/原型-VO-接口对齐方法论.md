# 原型-VO-接口对齐方法论

## 1. 核心认知

页面原型的唯一目的，是在编码开始前，让需求、交互、数据字段三者在同一张图上对齐。

原型图是交互意图的可视化。VO 设计是原型图的数据翻译。接口是 VO 在时序上的分解。

三者本质是同一个东西的三个视角：

```text
原型图：用户看到什么、怎么操作
VO：页面需要什么数据才能渲染
接口：这些数据如何读取、提交和拆分时序
```

因此顺序应该是：

```text
原型图 → VO 设计 → 接口文档
```

而不是原型、VO、接口分开设计，更不应该先从数据库字段正推页面展示。

## 2. 原型的约束力

原型不是可交付的前端代码，但它是产品需求的视觉表达。

在 AI-Native 开发中，原型对后续实现的约束力可以这样划分：

| 维度 | 约束力 | 说明 |
| --- | --- | --- |
| 页面流程与跳转 | 强约束 | 哪些页面存在、如何跳转，必须严格对齐 |
| 功能入口与操作 | 强约束 | 按钮、操作项、表单字段不能随意增删 |
| 展示字段 | 强约束 | 列表项、详情页展示哪些字段，VO 设计必须对齐 |
| 视觉样式细节 | 弱约束 | 颜色、间距、字体大小允许在实现时微调 |
| 动效与过渡 | 弱约束 | 原型中的动效仅作参考，实现时可简化 |
| 示例数据 | 参考 | 用于演示和接口示例校准，不直接等于真实数据 |

## 3. VO 如何设计

VO 以页面渲染的最小必要数据集为准，同时考虑复用边界。

基本原则：

- 原型中出现的每个字段，VO 里必须有对应字段
- 原型中没有出现的字段，VO 不应主动引入
- 列表页、详情页、管理页的 VO 不要混用
- 页面复用同一展示结构时，才抽公共 VO
- 公共 VO 只承载真正跨页面稳定复用的字段

典型判断：

- 收藏列表卡片只需要标题、来源、备注、保存时间、标签 chip，就不应该带详情页才需要的完整链接信息
- 标签管理页需要 `bookmarkCount`，但收藏卡片里的标签 chip 不需要，所以应区分完整标签 VO 和轻量标签 VO
- 用户资料页和登录态都展示用户基础信息，可以复用 `UserVO`

## 4. 接口如何拆分

接口不是简单把 VO 原样搬出来。接口要表达页面数据在时序上的读取和提交。

拆分时优先考虑：

- 页面是否需要并发加载多个独立区域
- 操作是否只修改局部字段
- 同一能力是否被多个页面复用
- 是否需要避免把某个页面的聚合结构扩散到全局

典型原则：

- 首页可以由统计、标签、最近保存三个小接口组合，不一定需要首页大聚合接口
- 搜索页和标签筛选页入口不同，但底层查询语义相同，可以共用搜索/筛选接口
- 编辑备注、调整标签、删除收藏是局部操作，不需要重新提交整个详情 VO
- 导入链接是一个任务流：先创建收藏，再补备注和标签

## 5. 交互即规则

原型中的交互分支不是装饰，而是业务规则的外显。

例如：

- 删除收藏需要确认，说明它是破坏性操作
- 删除标签只允许删除无关联标签，说明接口和数据库都要能表达这个约束
- 合并标签需要二次确认，说明这个操作会迁移关系并删除源标签
- 标签 chip 即时选择，说明它是低风险、可逆、所见即所得的操作

因此，原型展示了什么，接口文档的规则部分就应该覆盖什么。

## 6. AI-Native 场景下的价值

在 AI 主导编码的场景里，原型不是给人“看个大概”的图，而是 AI 编码的上下文边界。

原型承担四个职责：

- 视觉规格文档：AI 生成前端时直接对照页面结构
- 字段校验基准：AI 设计 VO 或组件 props 时不能自行扩展字段
- 需求对齐工具：人工验收 AI 产出时有明确参照
- 避免过度设计的边界：AI 不应在原型之外发明未展示的功能

最重要的校准链是：

```text
需求文档
  ↓ 校准业务场景、MVP、规则
产品原型
  ↓ 校准页面、字段、交互、空状态
VO / 接口文档
  ↓ 校准请求、响应、错误码、权限
数据库设计
  ↓ 校准字段来源、关联关系、索引
代码实现
  ↓ 校准接口行为、页面表现、测试结果
```

当发现不一致时，不要只改代码。应先判断是哪一层文档错了，再从源头修正。
