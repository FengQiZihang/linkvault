# 系统分享采集实验说明

## 1. 目标

本实验工程只做一件事：

- 在 Android 真机上通过系统分享打开应用
- 尽可能完整保留系统分享 Intent 中携带的原始数据
- 对比不同平台分享载荷的数据形式差异
- 过滤普通冷启动/热启动 Intent，避免非分享数据混入样本

本实验不做业务保存，不做卡片设计，不做后端联调。

## 2. 工程位置

实验工程目录：`share-intake-lab/`

关键文件：

- `share-intake-lab/nativeResources/android/AndroidManifest.xml`
- `share-intake-lab/common/shareCapture.js`
- `share-intake-lab/pages/index/index.vue`

## 3. HBuilderX 如何使用

### 3.1 导入工程

1. 打开 HBuilderX
2. 选择 `文件 -> 打开目录`
3. 选择仓库中的 `share-intake-lab` 目录
4. 等待 HBuilderX 识别为 uni-app Vue3 项目

### 3.2 为什么必须使用自定义调试基座

本工程依赖 `nativeResources/android/AndroidManifest.xml` 给 `io.dcloud.PandoraEntryActivity` 增加 `ACTION_SEND` 的 `intent-filter`。

HBuilderX 标准基座不会带上这份自定义 AndroidManifest 配置，因此系统分享列表里可能看不到本应用。

要让系统分享入口在真机上生效，必须先打一个 **Android 自定义调试基座**。

### 3.3 生成自定义调试基座

1. 在 HBuilderX 中选中 `share-intake-lab` 项目
2. 点击顶部 `运行`
3. 选择 `运行到手机或模拟器 -> 制作自定义调试基座`
4. 平台选 `Android`
5. 按 HBuilderX 引导完成云端打包
6. 打包完成后，将生成的 `android_debug.apk` 安装到手机

一般生成路径在：

- `share-intake-lab/unpackage/debug/android_debug.apk`

### 3.4 真机运行

1. 手机安装刚才生成的自定义调试基座
2. 手机开启 USB 调试
3. 回到 HBuilderX
4. 选择 `运行 -> 运行到手机或模拟器 -> Android 设备`
5. 看到首页出现“Android 系统分享采集实验室”即表示运行成功

## 4. 如何做系统分享测试

### 4.1 单次测试动作

1. 在目标 App 中找到一条可分享内容
2. 点击系统分享
3. 选择 `分享采集实验室`
4. 应用打开后，会自动尝试读取当前 Intent 并落一条本地记录
5. 页面上手动给这条记录标记来源平台
6. 如有需要，在“测试备注”中写下观察结果
7. 复制 JSON 保存到你的测试记录里

### 4.2 建议固定测试的四个平台

- B站
- YouTube
- X
- 微信公众号

### 4.3 建议对比的字段

- `snapshot.action`
- `snapshot.type`
- `snapshot.extraText`
- `snapshot.extraSubject`
- `snapshot.extraHtmlText`
- `snapshot.extraStream`
- `snapshot.dataString`
- `snapshot.extras`
- `snapshot.clipData`
- `snapshot.rawIntent`

## 5. 当前实现会保留哪些分享数据

程序当前会尝试保留以下内容：

- `Intent.action`
- `Intent.type`
- `Intent.scheme`
- `Intent.dataString`
- `Intent.package`
- `Intent.component`
- `Intent.flags`
- `Intent.categories`
- `android.intent.extra.TEXT`
- `android.intent.extra.SUBJECT`
- `android.intent.extra.HTML_TEXT`
- `android.intent.extra.STREAM`
- `android.intent.extra.TITLE`
- `android.intent.extra.REFERRER_NAME`
- `extras` Bundle 中所有可枚举键值
- `ClipData` 的描述信息和各 item 内容
- `Intent.toString()` 的原始字符串

当前仅在识别到分享型 Intent 时写入记录，判定依据包括：

- `action` 为 `android.intent.action.SEND`
- `action` 为 `android.intent.action.SEND_MULTIPLE`
- 或存在 `EXTRA_TEXT`、`EXTRA_SUBJECT`、`EXTRA_HTML_TEXT`、`EXTRA_STREAM`
- 或存在可枚举 extras / clipData

## 6. 数据保存方式

- 数据仅保存在本机 `uni.setStorageSync` 本地存储中
- 不会上传服务器
- 点击“清空全部”只影响当前设备本地记录

## 7. 可能遇到的现象

### 7.1 分享列表里没有本应用

优先检查：

- 你安装的是不是 **自定义调试基座**，而不是标准基座
- 是否重新安装过最新生成的 `android_debug.apk`
- 当前分享的 MIME 类型是否被目标 App 标成了非文本类型

### 7.2 应用打开了，但没有记录

优先检查：

- 当前是不是 `App-Android` 真机环境
- 目标 App 是否真的走了系统分享，而不是站内私有分享弹窗
- 某些机型在应用已经驻留后台时，需再次点击“重新读取当前分享”验证一次

### 7.3 某个平台分享内容不完整

这是本实验要观察的重点之一。原因可能包括：

- 平台本身只放了文案，没有放独立链接字段
- 平台把链接放在 `EXTRA_TEXT`
- 平台把内容拆到 `ClipData`
- 平台通过私有 extras 键传值

## 8. 推荐测试记录方法

每个平台至少保留 1 条记录，并把 JSON 复制出来单独归档。建议你最终整理成下面这种结构：

```text
B站
- 分享动作：视频详情页 -> 系统分享
- 记录ID：xxx
- 关键差异：xxx

YouTube
- 分享动作：视频详情页 -> Share
- 记录ID：xxx
- 关键差异：xxx

X
- 分享动作：帖子详情页 -> Share post via
- 记录ID：xxx
- 关键差异：xxx

微信公众号
- 分享动作：文章页 -> 系统分享
- 记录ID：xxx
- 关键差异：xxx
```
