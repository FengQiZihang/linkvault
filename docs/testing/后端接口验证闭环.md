# 后端接口验证闭环

## 1. 文档目的

本文用于约束 AI 或人工在完成后端接口开发后，不只停留在 `mvn test` 编译通过，而是必须启动 Spring Boot 服务，并通过真实 HTTP 请求验证接口行为。

适用范围：

- 后端 Controller / Service / Mapper 改动
- 接口路径、请求体、响应 VO、错误码改动
- 鉴权、用户隔离、标签、收藏、导入等业务接口改动

## 2. 验证分层

后端验证分三层：

1. 编译与单测：确认代码能编译，基础 Spring 上下文和已有测试不失败。
2. 启动验证：确认服务能真实启动，端口、数据库连接、拦截器和配置没有启动时错误。
3. 接口验证：确认接口能被 HTTP 调用，并返回符合接口文档的结构和业务结果。

`mvn test` 只覆盖第一层，不能替代启动验证和接口验证。

## 3. AI 标准执行顺序

每次涉及后端代码改动时，AI 默认按以下顺序执行。

### 3.1 编译与单测

```bash
cd backend/linkvault-server
mvn test
```

只有 `mvn test` 通过后，才进入数据库重置和接口验证。

### 3.2 重置测试数据库

启动后端前，必须先回到仓库根目录，按顺序执行：

```bash
cd /home/hang/workspace/AndroidAPP开发
mysql -ulinkvault -p -h 127.0.0.1 -P 3307 < docs/sql/001_schema.sql
mysql -ulinkvault -p -h 127.0.0.1 -P 3307 < docs/sql/002_seed_test_data.sql
```

说明：

- `001_schema.sql` 负责重建库表结构。
- `002_seed_test_data.sql` 负责写入接口验证所需的测试数据。
- 如果本地 MySQL 使用不同端口、账号或连接方式，按 `docs/environment/WSL开发环境配置记录.md` 中的本机配置调整命令。
- 这一步会重置本地 `linkvault` 测试库；不要在包含重要个人数据的数据库上执行。

### 3.3 启动后端

```bash
cd backend/linkvault-server
mvn spring-boot:run
```

服务启动后，在另一个终端执行必要的 `curl` 验证。

验证完成后必须关闭后端进程，避免端口 `8080` 被占用。

### 3.4 失败后的重试规则

如果接口测试失败，AI 必须先根据响应、日志、数据库状态定位原因并自行修正代码或文档，然后重新执行完整闭环：

```text
mvn test
  -> 执行 001_schema.sql
  -> 执行 002_seed_test_data.sql
  -> 启动 Spring Boot
  -> curl 接口验证
```

只有接口验证通过后，才能把结果交给用户复核。

验证通过后不要再清空数据库。测试过程中生成的数据应保留，方便用户继续用 Apifox 查看表数据和接口效果。

### 3.5 测试数据脚本调整原则

`docs/sql/002_seed_test_data.sql` 不是一次性写死的固定数据，而是服务接口验证和人工联调的测试资产。

当出现以下情况时，AI 可以提出修改 `002_seed_test_data.sql` 的建议：

- 新接口需要更完整的用户、标签、收藏、链接关系样本。
- 现有测试数据无法覆盖关键成功路径或失败路径。
- 现有测试数据与接口文档、数据库字段设计或代码实现不一致。
- Apifox 人工测试需要更容易观察的样本数据。

修改建议应说明：

- 为什么现有 seed 不够用。
- 需要新增或调整哪些数据。
- 会影响哪些接口验证。

如果用户同意，或当前任务本身要求补充测试数据，AI 应同步修改 `002_seed_test_data.sql`，并在最终说明中列出修改原因。

## 4. 基础连通性验证

健康检查：

```bash
curl -i http://localhost:8080/ping
```

预期：

- HTTP 状态码为 `200`
- 响应体为统一响应结构
- `data.message` 为 `pong`

旧路径不应作为后端路径存在：

```bash
curl -i http://localhost:8080/api/ping
```

预期：

- 直连后端时返回 `404`
- `/api` 只属于前端代理、Apifox 代理或网关对外暴露的前缀

## 5. 登录与鉴权最小闭环

开发环境验证码固定为 `123456`。每次要验证受保护接口时，先完成登录闭环。

发送验证码：

```bash
curl -i -X POST http://localhost:8080/auth/sms-code \
  -H 'Content-Type: application/json' \
  -d '{"phone":"13800000000"}'
```

登录并保存响应：

```bash
curl -s -X POST http://localhost:8080/auth/login \
  -H 'Content-Type: application/json' \
  -d '{"phone":"13800000000","code":"123456"}' \
  -o /tmp/linkvault-login.json
```

提取 token：

```bash
TOKEN=$(sed -n 's/.*"accessToken":"\([^"]*\)".*/\1/p' /tmp/linkvault-login.json)
```

验证当前用户接口：

```bash
curl -i http://localhost:8080/me \
  -H "Authorization: Bearer ${TOKEN}"
```

预期：

- HTTP 状态码为 `200`
- 响应体 `code=0`
- `data.phone` 与登录手机号一致

未携带 token 的受保护接口应失败：

```bash
curl -i http://localhost:8080/me
```

预期：

- HTTP 状态码为 `401`
- 响应体为统一错误结构

## 6. 标签接口最小闭环

当标签接口发生改动时，至少验证：

```bash
curl -i http://localhost:8080/tags \
  -H "Authorization: Bearer ${TOKEN}"
```

创建标签：

```bash
curl -i -X POST http://localhost:8080/tags \
  -H "Authorization: Bearer ${TOKEN}" \
  -H 'Content-Type: application/json' \
  -d '{"name":"AI测试标签","pinned":false}'
```

再次获取标签：

```bash
curl -i http://localhost:8080/tags \
  -H "Authorization: Bearer ${TOKEN}"
```

预期：

- 创建接口返回 `code=0`
- 返回对象包含 `tagId`、`name`、`pinned`、`bookmarkCount`
- 再次查询能看到新创建的标签

如重复执行会触发标签重名，AI 应说明这是测试数据复用导致的预期冲突，或改用新的标签名。

## 7. Apifox 与 AI 验证的关系

Apifox 是人工联调和观察接口效果的主要工具，但 AI 默认不能只说“可以用 Apifox 测”就结束任务。

AI 侧最低要求：

- 使用 `curl` 完成当前改动相关接口的最小成功路径验证
- 对鉴权、参数错误、重复提交、资源不存在等关键失败路径至少验证一类
- 在最终说明里列出实际请求过的接口路径和结果

用户侧 Apifox 验收：

- 使用 `docs/engineering/openapi.yaml` 导入接口
- 环境 URL 指向 `http://localhost:8080`
- 如果前端环境使用 `/api`，在代理层 rewrite 到后端资源路径
- 手写 JSON 数据测试成功路径和失败路径

AI 不应声称 Apifox 已通过，除非用户明确反馈 Apifox 测试结果，或当前环境提供了可执行的 Apifox CLI / 自动化集合。

## 8. 最终回复必须包含的验证信息

后端改动完成后，AI 最终回复至少包含：

- 已运行的命令，例如 `mvn test`
- 是否执行过 `001_schema.sql` 和 `002_seed_test_data.sql`
- 是否启动过后端服务
- 实际请求过的接口路径
- 每个关键请求的 HTTP 状态码和业务 `code`
- 测试通过后数据库是否已保留，便于用户继续检查
- 未验证的风险或需要用户用 Apifox 继续确认的点

如果因为数据库、端口、依赖、网络或环境问题无法完成接口验证，AI 必须说明阻塞原因，并给出用户可复现的下一步命令。
