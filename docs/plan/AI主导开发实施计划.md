# LinkVault AI 主导开发实施计划

## 1. 目标

本文档用于把 LinkVault 项目组织成“AI 负责绝大部分编码、文档维护、测试与联调；人主要负责环境配置和需求裁决”的开发模式。

当前仓库已经完成需求、技术栈、实践方法和原型层面的基础收敛，但尚未落地真实工程目录与可运行代码。因此本计划的核心目标不是继续扩写前置文档，而是建立一套可直接进入工程实施的阶段拆分方式。

适用范围：
- LinkVault Android MVP
- 前端 UniApp + Vue3
- 后端 Spring Boot 3 + Java 21
- MySQL 8
- Playwright 联调与端到端验证

---

## 2. 执行原则

### 2.1 先本地跑通，再分批扩展

不采用“先把所有基础设施铺满，再开始业务”的方式，而是先在本地建立最小可运行环境与工程骨架，优先跑通主链路；待本地开发流程稳定后，再引入 Docker 做环境隔离与部署支持。

### 2.2 文档、代码、测试同步推进

不采用“后端做完 -> 前端做完 -> 最后联调”的串行方式。每个业务切片都要同步完成：
- 后端接口
- 前端页面接入
- 自动化测试
- Playwright 联调
- 过程文档更新

### 2.3 过程文档由 AI 跟随实现维护

数据库文档、接口文档、测试清单、联调说明、实施进度等文档，不要求人工在开发前完全定义，而由 AI 在开发过程中基于真实代码与测试同步生成维护。

### 2.4 严格遵守 MVP 边界

当前实施必须遵守以下范围：
- 只做 Android MVP
- 只做 APP 内手动粘贴 URL 导入
- 不引入系统分享桥接作为当前前置任务
- 不引入 AI 总结、语义搜索、自动推荐等扩展能力

## 4. 总体阶段拆分

## 阶段 0：冻结执行基线

### 目标
把需求边界、技术约束、AI 协作方式整理成后续可直接驱动编码的唯一事实源。

### 输出
- 工程实施顺序明确
- AI 维护文档范围明确
- 人工职责边界明确

### 说明
当前仓库在这一阶段已经基本完成，因此后续不再重复扩写抽象前置文档，而直接进入工程实施。

---

## 阶段 1：本地最小可运行环境与工程骨架

### 目标
把当前文档型仓库升级为真正可启动、可扩展、可测试的工程仓库，并优先在本地跑通开发链路。

### 核心产物
- `backend/`：Spring Boot 单体单模块工程
- `frontend/`：UniApp Vue3 工程
- `tests/e2e/` 或根目录 Playwright 配置
- 根目录：`.env.example`、`README.md`

### 执行策略
第一版优先完成：
- 后端本机运行
- 前端本机运行
- 本地数据库连接方案跑通
- Playwright 本机运行

说明：
当前阶段不引入 Docker 作为前置依赖，优先降低理解与调试复杂度，先确保本地主链路可快速迭代。

---

## 阶段 2：数据模型与接口契约先行

### 目标
在大量写业务代码前先固化数据库结构和 API 契约，降低 AI 生成代码时的漂移风险。

### 核心产物
- `docs/sql/001_init.sql`
- `docs/database/数据库设计.md`
- `docs/api/openapi.yaml`
- `docs/api/接口说明.md`
- 后端 `application*.yml`

### 推荐最小实体
- `lv_user`
- `lv_sms_code`
- `lv_link`
- `lv_bookmark`
- `lv_tag`
- `lv_bookmark_tag`

### 必须遵守的固定约束
- API 前缀：`/api/v1`
- 鉴权方式：JWT Bearer
- 响应结构：`code/msg/data`
- 分页参数：`page/pageSize`
- 表名前缀：`lv_`

---

## 阶段 3：账号 + 导入保存主链路

### 目标
优先打通第一个可验证业务闭环：登录后成功保存链接。

### 后端模块
- 验证码发送/登录
- JWT 下发与鉴权
- 首次注册昵称设置
- 手动粘贴 URL 保存
- 同用户同链接去重
- 元信息异步抓取基础框架

### 前端模块
- 登录页
- 首次注册页
- 导入页
- 保存成功反馈

### 验证
- 后端登录/鉴权/导入保存自动化测试
- Playwright 第一条主链路：登录 -> 粘贴 URL -> 保存成功

---

## 阶段 4：备注 + 标签绑定闭环

### 目标
把“保存链接”升级为“带上下文的可回找收藏记录”。

### 后端模块
- bookmark 备注新增/编辑
- 标签创建
- 多标签绑定/修改
- 未打标签状态支持

### 前端模块
- 导入承接页备注输入
- 标签快速点选
- 即时新增标签
- 详情页备注/标签编辑
- 首页最近保存与标签入口

### 验证
- 后端备注、标签绑定规则测试
- Playwright：导入 -> 写备注 -> 新建标签 -> 多标签绑定 -> 详情验证

---

## 阶段 5：搜索 + 详情 + 回找闭环

### 目标
验证 MVP 是否真正完成“找得回”。

### 后端模块
- 关键词搜索（标题/备注）
- 标签筛选
- 关键词 + 标签叠加筛选
- 详情查询
- 删除收藏

### 前端模块
- 搜索页
- 结果页
- 详情页
- 打开原始链接

### 验证
- 后端查询规则测试
- Playwright：造数据 -> 搜索命中 -> 标签交集筛选 -> 详情验证

---

## 阶段 6：标签管理高级规则

### 目标
补齐标签重命名、合并、删除等高风险业务规则。

### 后端模块
- 标签重命名
- 标签合并
- 标签删除约束
- 数据一致性处理

### 前端模块
- 标签管理页
- 合并/删除确认交互

### 验证
- 后端标签规则自动化测试
- Playwright 标签管理主流程回归

---

## 阶段 7：Docker 化、全量回归与可交付化

### 目标
在本地开发链路稳定后，引入 Docker 做环境隔离与部署支持，并把项目从“能跑”升级为“可重复启动、可重复验证、可持续维护”。

### 核心产物
- Docker 环境配置
- 可运行的 README
- 完整测试清单
- E2E 场景清单
- 联调说明
- 实施进度记录

### 验证范围
1. 本地启动：前端与后端都可运行
2. 自动化测试：核心业务规则通过
3. Playwright：登录 -> 导入 -> 备注 -> 标签 -> 搜索 -> 详情 主链路回归
4. Docker：数据库或完整环境可成功拉起，并支撑部署/复现
5. Android：APK 或真机验证 APP 内手动粘贴 URL 主链路

---

## 5. 推荐模块开发顺序

建议按以下顺序推进：

1. 环境与工程骨架
2. 账号认证
3. 链接导入与保存
4. 备注
5. 标签基础能力
6. 首页与最近保存
7. 搜索与筛选
8. 详情页
9. 标签管理高级规则
10. 元信息异步抓取增强
11. 全量回归与验收

这一顺序的原则是：
- 先打通最短价值路径
- 再补回找能力
- 最后补复杂规则与增强项

## 7. 前后端、测试、联调穿插方式

每个业务切片都应按以下顺序推进：

1. 后端接口与自动化测试先落地
2. 前端立即接入并实现对应 UI
3. 立刻补充 Playwright 对应联调用例
4. 同步更新 OpenAPI、SQL、测试清单、联调说明

也就是说，每个切片都必须形成一个：
- 可运行
- 可联调
- 可验证
- 可记录
的增量，而不是只完成单侧开发。

## 8. 验证策略

## 8.1 环境验证
- 本地数据库连接方案可运行
- 后端可通过配置连接数据库
- 在后续 Docker 化阶段，`docker compose up -d` 可成功拉起目标环境

## 8.2 后端验证
- 健康检查接口可访问
- 登录、鉴权、导入、备注、标签、搜索、详情相关测试通过

## 8.3 前端验证
- UniApp H5 调试可运行
- 登录页、导入页、搜索页、详情页可正常交互
- UI 风格逐步对齐原型与设计目标

## 8.4 联调验证
- 使用 Playwright CLI + playwright skill 执行主链路回归
- 每完成一个切片就新增对应 E2E，而不是最后集中补

## 8.5 终验验证
- 新环境可按 README 启动
- 核心自动化测试通过
- Playwright 主链路回归通过
- Android 真机 / APK 至少验证手动粘贴 URL 导入主链路

在本地骨架落地后，立即进入阶段 2，固化 SQL 与 OpenAPI，再开始主链路开发；待本地流程稳定后，再推进 Docker 化。
