# LinkVault 原型设计说明

## 1. 原型定位

本原型是产品需求的视觉表达，不是可交付的前端代码。

它的唯一目的是：在编码开始前，让需求、交互、数据字段三者在同一张图上对齐。

## 2. 原型与真实前端的关系

| 维度 | 原型的约束力 | 说明 |
| --- | --- | --- |
| 页面流程与跳转 | **强约束** | 哪些页面存在、如何跳转，必须严格对齐 |
| 功能入口与操作 | **强约束** | 按钮、操作项、表单字段不能随意增删 |
| 展示字段 | **强约束** | 列表项、详情页展示哪些字段，VO 设计必须对齐 |
| 视觉样式细节 | **弱约束** | 颜色、间距、字体大小允许在实现时微调 |
| 动效与过渡 | **弱约束** | 原型中的动效仅作参考，实现时可简化 |
| 示例数据 | **参考** | 原型中的数据用于演示，接口文档的示例数据应与之对齐 |

## 3. 设计原则

### 3.1 字段不多不少

原型中出现的每个字段，接口文档的 VO 里必须有对应字段。原型中没有出现的字段，VO 不应主动引入（避免前端处理不需要的数据）。

### 3.2 数据驱动展示

原型中的数字、文案、标签数量都是有意设计的，反映了真实使用场景。接口文档的响应示例应直接复用原型中的示例数据，保持一致性，便于 AI 编码时交叉对照。

### 3.3 交互即规则

原型中的每一个交互分支（弹窗、确认框、错误状态、空状态）都对应一条业务规则。原型展示了什么，接口文档的规则部分就必须覆盖什么。

## 4. 在 AI 主导编码场景下的使用方式

本项目由 AI 负责实际编码，人工负责需求定义与边界把控。原型在这个模式下承担以下职责：

- **视觉规格文档**：AI 生成前端代码时，原型是唯一可以直接对照的视觉参考
- **字段校验基准**：AI 设计 VO 或组件 props 时，以原型展示字段为准，不自行扩展
- **需求对齐工具**：人工在验收 AI 产出时，以原型为基准检查功能完整性
- **避免过度设计的边界**：AI 不应在原型之外自行发明未展示的功能或字段

## 5. 原型与文档的对齐顺序

正确顺序：**原型图** → **VO 设计** → **接口文档**

不应该先写 VO 再对照原型。VO 字段必须从原型的展示需求反推，而不是从数据库字段正推。

## 6. 原型文件说明

| 文件 | 说明 |
| --- | --- |
| `产品原型.html` | 可交互 HTML 原型，包含全部页面和页面跳转逻辑，直接用浏览器打开 |
